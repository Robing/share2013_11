前端工程师的基本职责：还原设计还原"活"的设计。

"活"的设计(Living Design, 简称LD)?

首先，产品是"活"的, 特点: 扩展和进化速度快。入囗多、需求变化快、产品需要覆盖各种设备和平台

快速、持续进化的产品，举例:

  • 未登录首页：2个版本
  • 登录首页: 最多时6个版本
  • 小组首页：最多时5个版本
  • 广播类型全集：195种
  • 提醒类型全集：67种
  • 所有新功能上线都要经过对比测试
  • 产品实现的特点：展现形式多、新旧代码并存、业务逻辑复杂、迭代快、需求多变、兼容移动设备...
  • 产品实现的挑战：产品设计和技术的紧密协作，体现为系统的设计和高灵活性的前端架构

传统合作模式(按页面交付) > 外包模式 > OUT!

So, "活"的产品需要"活"的设计

对"活"的理解

还原设计，不是还原设计稿！设计稿是“死”的、文档是“死”的，简单实现一个“死”的东西毫无价值

生命体基本特征的类比：

  1. 体内平衡 ············ 这种自动调节指适配性。适配内容变化(文字长度、图片尺寸)、屏幕等各种情况
  2. 组织性  ············ UI可解构为UI元素的系统构成(如原子设计、卡片设计)
  3. 新陈代谢 ············ 新的UI元素不断替代旧的元素。保持整体设计的活力
  4. 生长   ············ 从最小可用形态(MVP)，跟随需求、不断迭代、持续完善
  5. 适应性  ············ 能适应用户的使用环境(如响应式设计)
  6. 反应性  ············ 能响应用户的各种操作，反馈各种状态
  7. 繁殖   ············ 基于现有的生态体系，快速孵化新产品

定义：LD是由有组织的UI元素构成，能适应不同的使用环境，响应用户的行为，灵活适配内容，可以扩展和继承的设计。

LD的基本特征: 1.组织性、2.扩展性、3.适配性、4.响应性。

比较传统设计(契约-交付式设计): 单一vs.多态，固定vs.变化，颠覆vs.进化

实现LD的前提

  1. 设计师和前端工程师是协作关系：关注点不同，为了同一个目标 -- 实现一个“活”的产品
  2. 传统产品实现流程的转变
  3. 传统设计实现思路的转变

还原LD - 再造协作流程

案例分析: A项目(项目名称不便透露)的协作过程。

  1. PM写了一份简要的PRD，PM x Designer x Engineer Review
  2. 确定M1的目标，M1定义了MVP
  3. 根据M1，Designer出Wireframe，PM x Designer Review
  4. 根据M1，FE x BE出第1版原型。(有测试数据，无样式，尽可能少写JS/CSS，同步的流程)
  5. 基于原型，VD x ID开始系统设计(前面所述)。分块递交。PM能看到“生长”过程
  6. M1的开发着重架构设计(基于组件的思想)，分层、分块实现
  7. 不陷在某个复杂的设计实现里，前面所述的迭代设计
  8. 阶梯式进展
  9. 好的协作：多种角色间，不断抛出问题，解决问题的过程

还原LD - 组件化的静态文件管理系统(2)

一个静态文件管理系统的"刚需"：

  1. 实现组件化 (想想完整组件的含义)

  3. 自动打包
  4. 部署方式
  5. 支持CSS的预处理(SCSS/LESS/Stylus)

响应式设计(Responsive Web Design, WD)不仅仅局限于响应屏幕。本质上更好适应用户的使用环境。

高级feature: 原生MultipleUpload, DragDrop, Paste, 环形进度条
